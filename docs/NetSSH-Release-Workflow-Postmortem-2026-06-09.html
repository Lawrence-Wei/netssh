NetSSH Release Workflow 复盘总结（2026-06-09）

目标：把 Release 工作流从“经常失败”调整为“可稳定自动产出发布文件到 GitHub Release”。

1. 背景

仓库在 Release 工作流运行时，Linux 构建和依赖解析问题频发，导致上传环节无法完成。用户要求能自动编译并上传，且要求后续可控。

2. 问题与根因

问题 A：Linux arm64 apt 404

  • 现象：security.ubuntu.com 上出现 arm64 package 404。
  • 根因：runner 的源组合与架构匹配不一致。
  • 影响：arm64 job 失败，release 链路被阻塞。

问题 B：源文件格式导致 apt 解析错误

  • 现象：Malformed stanza ... (type)
  • 根因：自定义源文件写法与解析格式不匹配。
  • 影响：即便重试也会被源解析阶段直接终止。

问题 C：create-release 依赖过于严格

  • 现象：Linux job 失败时，create-release 被跳过。
  • 根因：needs 同时依赖了已移除/高风险平台。
  • 影响：用户看到“前面都成功”但发布没发生。

3. 实施方案与结果

  • 先尝试修复 arm64 依赖与源配置，但在格式兼容上仍有难点。
  • 按用户要求：移除 Linux 版本构建。
  • 删除 build-linux-x64 与 build-linux-arm64。
  • 将 create-release 的依赖改为：Windows + macOS + npm publish。
  • 移除 Linux artifact 下载步骤。

4. 验证

  • 工作流 run（手动触发）：v1.1.12
  • Run ID：27195070216
  • 最终状态：success，且 Release 已创建。

5. 关键教训

  1. 发布链的关键依赖要谨慎。不可让“可选/不稳定平台”阻塞全部流水线。
  2. Linux cross build 的源配置要先在隔离环境验证，避免在主发布链里反复试错。
  3. 工作流改完后先看 jobs 状态树，若 create-release skipped，直接回看上游 needs。

6. 未来建议

  • 若需要恢复 Linux，建议新建独立可选 workflow，不作为主 release 的硬依赖。
  • 保留当前主流程的高稳定平台：Windows/macOS/npm。
